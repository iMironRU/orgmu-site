# Сначала ищи макет — единое правило

**Перед тем как верстать страницу, посмотри `design-ref/pages/`. Макет,
скорее всего, уже есть.** Имена английские, страницы русские — `Приложения
вуза` лежат в `Apps.dc.html`, карточка приложения в `AppDetail.dc.html`,
места — в `Locations.dc.html`. Не найдя по прямому переводу, грепните по
заголовку:

    grep -rl "Приложения вуза" design-ref/pages/

Так уже вышло: страницы приложений сделаны мимо `Apps.dc.html` и
`AppDetail.dc.html`, хотя те лежали в репозитории с первого дня. Получился
свой дизайн — и его пришлось переделывать.

**Шапка страницы — синяя плашка** `bg-brand text-white` во всю ширину, над
контентом, внутри `max-w-[1146px]`: хлебные крошки, `<h1>`, лид. Так во всех
макетах и на 18 страницах сайта. Белая шапка с синим заголовком — расхождение,
а не вариант. Эталон — `src/app/mesta/page.tsx`.

Исключение: витрины и карточки новостей, известий, мероприятий и персон — у
них своя шапка по своим макетам (`News.dc.html`, `Article.dc.html`,
`StaffMember.dc.html`).

# Ссылки и язык — единое правило

**Внутренние ссылки ставятся только через `@/components/Link`. Импортировать
`next/link` напрямую нельзя.**

Наш `Link` — это `next/link` плюс одно знание: какой язык выбран. Он сам
добавляет префикс (`/en/…`, `/kk/…`) там, где перевод раздела существует, и не
добавляет там, где его нет (иначе получили бы 404).

Почему это правило есть. Ссылок в проекте больше 80 в 47 файлах, и каждая
написанная «руками» через `next/link` молча выбрасывала человека на русскую
версию: выбрал английский, нажал на логотип — и снова русский сайт. Чинить их
по одной бессмысленно: следующая новая ссылка ломает всё заново. Поэтому знание
о языке живёт в одном месте, а не в каждом вызове.

Что нужно помнить при написании кода:

- **`<Link href="/…">`** из `@/components/Link` — все внутренние переходы.
- **`asset("/файл.pdf")`** — когда нужен именно `<a>` или CSS-`url()`.
- **Обычный `<a>`** — только внешние адреса, `#`, `mailto:`, `tel:`.
- **Язык живёт только в адресе.** У каждой русской страницы есть языковая пара
  (`/en/…`, `/kk/…`) — зеркало, которое рендерит тот же компонент. Там, где
  перевода ещё нет, содержимое остаётся русским, а сверху появляется полоса
  «доступно только на русском».
- Новый маршрут — заведите и зеркало в `src/app/[lang]/<путь>/page.tsx`: оно
  импортирует русскую страницу и берёт у неё же список адресов. Без зеркала
  ссылка на этот раздел выбросит человека из его языка.

**Проверка:** `grep -rn 'from "next/link"' src` — должен находить единственный
файл, сам `src/components/Link.tsx`. Любое другое попадание — будущая утечка
языка.

# Перевод страницы — единое правило

**Страница не переводится сама собой.** Языковое зеркало
(`src/app/[lang]/…/page.tsx`) просто рендерит русский компонент. Чтобы раздел
заговорил по-английски, нужно четыре вещи — все четыре, иначе получится
наполовину переведённая страница:

1. **Страница принимает `lang`** и переводит ДАННЫЕ через `translateData`,
   а свои строки — через `t()`. Зеркало передаёт `lang` в неё.
2. **Подписи клиентских компонентов живут в `src/lib/i18n/ui-defs.ts`.**
   Не в самом компоненте! Компонент помечен `"use client"`, и Next подменяет
   экспорты такого модуля ссылками на клиентский чанк: серверная страница
   получит не объект, а прокси, `uiStrings()` вернёт пустоту, и подписи молча
   останутся русскими. Ловил это уже дважды — выглядит особенно обманчиво,
   когда данные рядом переведены.
3. **Плашка `TranslationNotice`** на языковой версии — человек должен видеть,
   что перевод машинный, и иметь ссылку на оригинал.
4. **Адрес добавлен в `translatedPaths`** (`src/app/layout.tsx`), иначе поверх
   переведённой страницы висит полоса «доступно только на русском».

**ФИО не переводят, а транслитерируют** (`lib/i18n/translit.ts`): машина
делает из «Кузнецова» Smith. Ключ `fio` исключён в `rules.json`. На казахском
ФИО остаются кириллицей — язык её и использует.

**Ключи фильтров считаются ДО перевода.** Фильтры педсостава сопоставляют
должность и степень по русским подстрокам («доцент», «доктор»); на
переведённых данных они молча дадут пустой список. Считайте ключ на сервере и
передавайте отдельным полем.

**Данные, которые собираются кодом** (структура, персоны), попадают в перевод
через `scripts/i18n/export-strings.ts` — он берёт строки у тех же загрузчиков,
что и сайт. Не повторяйте разбор в скрипте перевода: вторая реализация
разойдётся с первой.

**Что намеренно остаётся русским:** юридические значения sveden, названия
документов рядом со ссылкой на русский файл, аббревиатуры баз 1С. Это записано
в `rules.json`, а не разбросано по коду.

# Ссылки и basePath — единое правило

**Внутренний адрес (`/novosti`, `/app-launcher/buh`, `/files/…`) ставится
только через `<Link>` или `asset()`. Обычный `<a href="/…">` ведёт в 404.**

Сайт живёт на GitHub Pages по адресу `/orgmu-site`, и Next подставляет
basePath **сам — но только в `next/link` и `next/image`**. В обычном `<a>`,
в `style={{ backgroundImage: url(...) }}`, в `<iframe src>` он не появится:
получится `/novosti` вместо `/orgmu-site/novosti`, то есть 404 на проде.
Локально при пустом basePath это НЕ видно — ошибка вылезет только после
деплоя.

Ловил себя на этом пять раз: фавикон, баннер на главной, файлы в карточках,
кнопки приложений, карта сайта (9 ссылок сразу).

- **`<Link href="/…">`** — внутренние переходы. Работает само.
- **`asset("/файл.pdf")`** — когда нужен именно `<a>` или CSS-`url()`.
- **Обычный `<a>`** — только для внешних адресов (`https://…`), `#`,
  `mailto:`, `tel:`.
- Ссылка может быть и внутренней, и внешней (лаунчер собирается для сайта и
  для внутренних хостов) — тогда развилка:
  `href.startsWith("http") ? <a> : <Link>`.

**Как проверить перед коммитом** (только так это и ловится):

    NEXT_PUBLIC_BASE_PATH=/orgmu-site npm run build
    grep -roE 'href="/(novosti|sveden|programmy|…)[/"]' out --include=index.html

Пусто — чисто. Есть попадания — это будущие 404.

# Файлы и документы — единое правило

**Любой файл на сайте выводится через `DocCard` / `DocCards`
(`src/components/sveden/`). Своей вёрстки карточки файла быть не должно.**
Это уже случалось трижды: каждый раз получался «почти такой же» вид, и его
приходилось переделывать.

Карточка даёт: цветной бейдж формата, название, дату, размер, иконку
скачивания, у заголовка группы — счётчик файлов.

Исключение одно: компактная строка-ссылка внутри чужой плитки (например,
«Положение» в карточке подразделения в `StructureView`). Это не список
документов — полная карточка там неуместна.

## Как это устроено

- **Формат** — из расширения в адресе. Не хардкодить: файл может быть `.doc`,
  `.rtf`, `.xlsx`.
- **Дата и размер** — файлы лежат на orgma.ru, их метаданные знает только
  сервер источника. Собираются HEAD-запросами:
  - типовые страницы (`content/pages/**`) → `node scripts/pages/enrich-files.mjs`
    → `content/pages/file-meta.json`;
  - sveden → `node scripts/sveden/enrich-docs.mjs` → `content/sveden/doc-meta.json`.
  Результат коммитится: сборка на GitHub Pages в сеть не ходит.
  **Добавили файлы — прогоните скрипт**, иначе карточки будут без даты и размера.
- **Длинные названия.** Названия на orgma — это полные заголовки актов (до 900
  символов). `DocCard` режет их по первой «ёлочке»: голова (тип, дата, номер)
  видна всегда, название в кавычках свёрнуто до двух строк и раскрывается
  кнопкой. Кнопка раскрытия — **рядом со ссылкой, не внутри неё**: вложенные
  интерактивные элементы ломают клавиатуру и скринридеры.
- **Адреса файлов.** В yml держим читаемыми (пробелы и кириллица), кодирование —
  при рендере (`encodeFileHref`). Не кодировать в данных: легко получить
  полу-кодированный гибрид и двойное `%2520`.
- **Ссылки внутренние (`/files/...`)** — обычный `<a>`, basePath автоматически
  не применяется. Прогонять через `asset()`.
- **ЭЦП не показываем.** В макете плашка «Подписан ЭЦП» есть, но orgma не
  публикует подписи (`.sig`) — подтвердить нечем. Рисовать её было бы враньём.
- **Нет файла — не прячем строку.** `DocCard` показывает заготовку: серый бейдж,
  «—» вместо размера, без ссылки.

# Данные в YAML

Строка с **двоеточием и пробелом** в незакавыченном скаляре ломает YAML: в
списке она молча превращается в объект и роняет рендер. Такие значения
закавычивать. Загрузчики (`pages.ts`, `events.ts`) на это ругаются с указанием
файла — читайте сообщение, оно точное.

# Данные — только реальные

Не выдумывать данные и не переносить демо-значения из макетов (там вымышленные
ставки, зарплаты, номера закупок, ФИО). Нет данных — заготовка с прочерками и
комментарием в yml, кто заполняет. Ссылки на файлы проверять: на orgma
встречаются битые.
